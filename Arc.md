INTRODUCTION AND PROJECT GOALSЦель данной технической спецификации — проектирование архитектуры для текстовой RPG-песочницы с открытым миром, глубокой социальной симуляцией и интеграцией LLM (ИИ).В версии 2.0 фокус смещен на стабильность в Production-среде, детерминизм симуляции, управление расходами на API и жесткий контроль над генерацией (отсечение галлюцинаций ИИ).1.1. Key Architectural PrinciplesПарадигма "Скелет и Мясо" (Strict Isolation): Жесткое разделение математической логики (Скелет - C# ядро) и нарративного оформления (Мясо - AI API).Двухуровневая генерация (Macro & Micro): Контент дестандартизирован. Общие структуры (Macro) генерируются в фоне, диалоги (Micro) — JIT.Zero-Trust AI & Fallback First: Система изначально предполагает, что API может вернуть мусор, упасть или уйти в таймаут. Для каждого ИИ-запроса есть оффлайн-заглушка.Thread-Safe Determinism: Фоновая симуляция полностью изолирована от UI-потока. Для исключения состояний гонки (race conditions) используется система снимков (Snapshots).2. FILE STRUCTURE AND ORGANIZATION/Core               # Системное ядро (C#)

3. CORE MODULE & EDGE CASES (ЯДРО И СИМУЛЯЦИЯ)3.1. Теневая симуляция (Thread Safety & Determinism)Проблема: UI поток читает данные NPC, пока фоновый поток симуляции их перезаписывает (Race Condition).Решение (Double-Buffering / Snapshots): Симуляция работает с GameState_Live. Раз в секунду симуляция блокируется (через Mutex или ReaderWriterLockSlim), делает глубокую копию стейта в GameState_Snapshot и отпускает блокировку. UI и рендер Godot читают данные только из Snapshot-а.Детерминизм: Все RNG (Random Number Generators) в симуляции используют фиксированный Seed, привязанный к ID мира. Это позволяет локализовать баги и покрыть симуляцию тестами.3.2. Сохранение и Консистентность (Save & Migrations)Версионирование контента: Каждое сохранение имеет SaveVersion и хэш WorldSeed.Миграции (Migrations): Если игра обновляется (добавили новый стат), при загрузке старого сейва отрабатывает скрипт миграции, который добавляет новые колонки в таблицы SQLite и заполняет их дефолтными значениями.Консистентность генерации: Скелеты (JSON), сгенерированные ИИ, перед записью в БД проверяются на целостность внешних ключей. Нельзя сохранить квест, который ссылается на удаленного NPC.4. AI MANAGEMENT (ПАМЯТЬ И КОНТЕКСТ)Управление памятью ИИ критично для экономии денег и предотвращения потери логики. Модуль ContextManager использует Слайдовое Окно Контекста (Sliding Window) со строгой приоритизацией.4.1. Приоритеты сборки Prompt (Token Allocation)На каждый запрос выделяется хард-лимит (например, 4000 токенов контекста). Данные пакуются по убыванию приоритета:System Rules (P0 - Несгораемые): ~300 токенов. Жесткие правила ("Ты в мрачном фэнтези, не ломай 4-ю стену, отвечай от лица торговца").Current Objective & Constraints (P1): ~200 токенов. Текущая цель движка (например, "Игрок провалил проверку харизмы, откажи ему").World & NPC State (P2): ~500 токенов. Локация, время суток, травмы игрока, статы NPC.Summary of History (P3): ~500 токенов. Сжатая выжимка старых диалогов с этим NPC (генерируется фоновым дешевым запросом после каждого диалога).Recent Chat History (P4 - FIFO): Оставшиеся токены (~2500). Последние 5-10 сообщений слово в слово. Если лимит превышен, самые старые вытесняются в Summary (P3).4.2. Контентные ограничения (Tone Control & Filters)JSON Structured Outputs: ИИ обязан отвечать в строгом формате, где есть поле character_thoughts (внутренний монолог для настройки тона) и spoken_text.Фильтр (ContentFilter.cs): Регулярные выражения (Regex) и списки стоп-слов локально проверяют текст на: современный сленг ("Окей", "Бро", "Компьютер"), попытки ИИ выдать себя за ИИ ("Как языковая модель..."). При срабатывании — авто-ретрай с системным предупреждением.5. PRODUCTION & ERROR HANDLING (ОШИБКИ И СТАБИЛЬНОСТЬ)5.1. AI Error System (Fallback & Retry)Модуль ErrorManager реализует паттерн Circuit Breaker.Retry & Rate Limit: Если API вернуло ошибку 429 (Too Many Requests) или 500, система ждет 1 сек -> 2 сек -> 4 сек (Exponential Backoff). Максимум 3 попытки.Timeout Handling: Жесткий таймаут на ответ — 10 секунд. Игрок не должен смотреть на зависший экран.Offline Mode (Деградация): Если все ретраи сгорели (API лежит) или у пользователя кончился интернет, LLMClient переключается в Offline.Действие: Игра не крашится. DialogBox вытаскивает из базы данных (SQLite) дежурную фразу, соответствующую стейту. Пример: вместо уникальной причины отказа ИИ, игра выдает хардкодный текст из БД: "Торговец презрительно отворачивается, отказываясь говорить с вами."5.2. Безопасность и Стоимость (Cost & Keys)Защита ключей: API ключи пользователя никогда не хранятся в логах или в открытом виде (PlainText). Они шифруются (через системный Keyring/DPAPI или локальный AES) в файле user_prefs.cfg.Cost Tracker: Каждое сообщение проходит через локальный токенизатор (например, порт TikToken). В UI (в меню настроек) выводится статистика: "Потрачено за сессию: ~15,000 токенов (≈ $0.15)". Для защиты от спама есть настраиваемый дневной лимит трат.6. DEVELOPMENT PIPELINE & TESTINGРазработка ведется итеративно, чтобы не закопаться в бесконечной симуляции.6.1. Pipeline ЭтапыФаза 1: Vertical Slice (MVP). * Цель: Доказать жизнеспособность связки Godot <-> LLM.Наполнение: 1 комната (Таверна), 1 NPC, базовые статы игрока. Боевка отключена. Теневая симуляция отключена. Тестируется PromptBuilder, скорость API, JIT-генерация текста и UI.Фаза 2: Core Loop (Alpha).Цель: Добавить математику и Скелет.Наполнение: Система инвентаря, квесты (графы State Machine), d20 боевка, переходы между локациями. Базовая теневая экономика для 3-х городов.Фаза 3: Full Sandbox (Beta).Цель: Масштабирование. Подключение Macro-генерации мира, экспорта префабов, сложной политики и глобальных кризисов.6.2. Тестирование (Testing Strategy)Система слишком сложна для ручного тестирования.Unit-тесты (Math & Core): Проверка боевых формул (Уворот, Урон), работы инвентаря, парсинга JSON-скелетов. Запускаются локально (без API).Simulation / Headless Тесты: Скрипт запускает теневую симуляцию без отрисовки графики со скоростью x1000. Проверяется, не сломается ли экономика спустя 10 игровых лет (нет ли переполнения переменных, не вымерли ли все NPC от случайного ивента).Load & Mock Тесты (API): Тесты LLMClient с использованием "Mock" (фейкового) сервера, который намеренно отдает ошибки 429, 500 и битые JSON, чтобы проверить устойчивость FallbackManager и оффлайн-режима.
