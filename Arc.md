1. INTRODUCTION AND PROJECT GOALS

Цель данной технической спецификации — проектирование архитектуры для текстовой RPG-песочницы с открытым миром, глубокой социальной симуляцией и интеграцией LLM (ИИ).

Данный документ представляет собой полную, развернутую архитектуру, фиксирующую протоколы общения с API нейросетей, детальную структуру хранения данных, жесткое ограничение на визуальное оформление, полный базовый справочник контента (Скелета), а также пошаговые инструкции по сборке сцен в редакторе Godot.

1.1. Key Architectural Principles

Парадигма "Скелет и Мясо" (Strict Isolation): Жесткое разделение математической логики и нарративного оформления.

Code-Only Rendering (Без внешних ассетов): На этапе разработки и в базовой версии игры полностью отсутствуют спрайты, текстуры, картинки и 3D-модели. Визуализация (карта, интерфейс, иконки) создается исключительно программным кодом Godot (использование примитивов ColorRect, Panel, RichTextLabel, процедурная генерация через _draw(), а также ASCII/UTF-8 символы для иконок).

Двухуровневая генерация (Macro & Micro): Контент генерируется в фоне (Macro) и JIT (Micro).

Zero-Trust AI & Fallback First: Защита от мусорных ответов и падений API.

Thread-Safe Determinism: Фоновая симуляция полностью изолирована от UI-потока.

Zero-Allocation Policy: Оптимизация Garbage Collector (C#).

Data-Driven (Отвязка контента от кода): Контент живет в JSON-файлах.

3. CORE MODULE & GODOT INTEGRATION

3.1. Теневая симуляция (Thread Safety)

Симуляция работает с GameState_Live. Раз в секунду делается копия в GameState_Snapshot под блокировкой ReaderWriterLockSlim. UI читает данные только из Snapshot-а.

3.2. Zero-Allocation

При формировании промптов конкатенация строк через + запрещена. Используются пулы и StringBuilder.

3.3. Настройка Autoloads (Синглтонов) в Godot

Для связи C# бэкенда с Godot и глобальной работы всех систем необходима строгая регистрация Autoloads в редакторе (Project -> Project Settings -> Autoload):

GameManager (узел из GameManager.cs). Важность: Метод _Ready() при старте игры автоматически вызывает DataManager.Initialize(), загружая все JSON-файлы (мобов, оружие, статы) в оперативную память.

Simulation (узел из Simulation.cs).

EventBus (узел из EventBus.cs).

3.4. Визуализация (Code-Only)

Отсутствие TextureRect с загруженными изображениями (png/jpg). Вся графика строится на темах Godot (Theme), узлах контейнеров и отрисовке линий.

4. ИГРОВЫЕ СИСТЕМЫ, ДАННЫЕ И МЕХАНИКА (THE SKELETON)

4.1. Боевая система (Stat-Driven Combat Framework)

Бой полностью изолирован от нейросети. Акцент смещен на детерминированные характеристики персонажа. Случайность дает лишь вариативность (±10% урона) и шанс критического удара/промаха.

Абстрактное позиционирование: Зоны Авангард (Vanguard) и Арьергард (Rearguard). Враг ближнего боя не может атаковать Арьергард, пока не уничтожен Авангард.

Очередь и тайм-менеджмент: Инициатива рассчитывается из стата Ловкости (DEX). На ход дается жесткий таймер (10 сек). Опоздание — авто-защита.

Отступление (Fleeing): Математическая проверка: DEX игрока + Кардио против DEX самого быстрого врага. Провал означает пропуск хода и удар в спину.

4.2. Состояние проигрыша (Death & Failure State)

Падение здоровья (HP) до 0 приводит к нарративному сдвигу и экономическому штрафу.

Механика Respawn: Simulation проматывает игровое время на 1-3 дня вперед. Персонаж "просыпается" в ближайшей безопасной зоне (Храм/Таверна).

Штрафы: Потеря % золота, резкое снижение прочности надетой экипировки, получение "Травмы" (дебафф статов).

Интеграция ИИ: При пробуждении LLMClient генерирует короткий текст о том, как вас спасли.

4.3. Износ экипировки (Durability)

В DataManager внедрены параметры CurrentDurability и MaxDurability.

Износ: Снижение прочности оружия при атаках (особенно по бронированным целям), брони — при получении урона.

Поломка: При достижении 0 предмет не исчезает, но его параметры обнуляются (до 1), пока он не будет восстановлен у кузнеца.

4.4. Глобальная симуляция: Королевства, Города и Маршруты

Мир — это взвешенный граф. Иерархия: Королевство -> Город -> Деревня -> Точка интереса (POI).

Генерация и Персистентность: При первой генерации любой локации (будь то королевство со своими городами или город со своими районами/зонами), алгоритм формирует внутреннее расположение дорог и маршрутов. Этот граф (сети дорог и позиционирование) жестко сохраняется в базу данных. При следующем визите игрок увидит ту же самую структуру маршрутов.

Случайные встречи: Во время путешествия по маршруту-дороге движок бросает кубики на Случайные Встречи (Random Encounters) на основе параметра Опасности дороги и Удачи игрока.

4.5. Система квестов и их выдачи (Quest Pipeline)

Сборка Скелета: QuestManager собирает математический граф. Пример: [Node A: Идти в Пещеру] -> [Node B: Убить Бандита] -> [Node C: Получить Золото].

Генерация Мяса: Скелет отправляется в ИИ. ИИ возвращает JSON с названием, лорной причиной и текстом для NPC.

Исполнение: После принятия квеста ИИ отключается. QuestManager локально следит за счетчиком прогресса.

4.6. Крафт и GOAP (Поведение NPC)

CraftingEngine: Валидирует наличие ингредиентов. Рассчитывает шанс успеха на основе Интеллекта. При успехе запрашивает у ИИ название и описание предмета.

GOAPEngine: Математически рассчитывает рутину NPC (Голод = 80 -> Цель: Поесть). Результат передается в ИИ как контекст для логичных ответов в диалоге.

5. AI MANAGEMENT & API INTERACTION (ВЗАИМОДЕЙСТВИЕ С ИИ)

Протокол общения с ИИ жестко стандартизирован. Игра ожидает только валидный JSON.

5.1. Структура API запроса и ответа

Запрос (User Payload):

{
  "context": {
    "location": "Таверна",
    "npc_state": {"name": "Гаррик", "profession": "Трактирщик", "goap_goal": "Обслуживать клиентов"},
    "player_action": "[Убеждение] Налей мне эля бесплатно."
  }
}


Ожидаемый ответ (JSON Schema):

{
  "thoughts": "Игрок манипулирует. Репутация высока. Я соглашусь.",
  "spoken_text": "Держи кружку. Но это в последний раз!",
  "action_triggers": [
    {"type": "give_item", "id": "drink_ale", "amount": 1}
  ]
}


5.2. Защита от Prompt Injection (InputSanitizer.cs)

Санитаризация: Ввод прогоняется через blacklists и RegEx (вырезаются мета-команды ignore instructions).

System Prompt Armor: Системные инструкции (P0) требуют игнорировать команды из блока Player_Input.

Валидация Скелета (Абсолютная защита): Если ИИ в action_triggers укажет выдать 10000 золота, GraphValidator отклонит этот ответ, так как он нарушает локальные лимиты узла.

5.3. Приоритеты Prompt и Долгосрочная память (RAG)

Распределение хард-лимита токенов:

System Rules (P0): ~300 токенов. Жесткие рамки.

Current Objective & GOAP (P1): ~200 токенов. Математическая цель NPC или квеста.

World State (P2): ~500 токенов. Время суток, травмы, погода.

RAG Context / Память (P3): ~600 токенов. VectorDB делает поиск старых семантически похожих фактов.

Recent Chat (P4 - FIFO): Оставшиеся токены на прямой лог беседы.

5.4. Настройка Ключей и API (LLMClient)

Для генерации лора и ответов требуется класс-обертка, инстанцирующий LLMClient. Временно или через UI настроек необходимо реализовать и вызвать метод установки учетных данных:
SetCredentials("ВАШ_КЛЮЧ", "https://api.openai.com/v1/chat/completions").

6. DATA PERSISTENCE & STORAGE ARCHITECTURE (ХРАНЕНИЕ ДАННЫХ)

6.1. Пользовательские директории (Godot user://)

user://saves/ — БД сохранений.

user://cache/ — Кэш ИИ.

user://exported_worlds/ — Префабы (Скелеты макро-мира) в формате JSON.

user://config/ — Настройки UI, ключи (зашифрованные).

6.2. База данных сохранений (SQLite)

Каждое сохранение — это файл .db. Позволяет подгружать мир чанками.

PlayerState: Базовые статы, уровень, координаты, время.

Inventory: Таблица связи [player_id] -> [item_id] -> [quantity] -> [current_durability].

WorldNodes: Состояние открытых локаций, включая их внутреннюю топологию и сгенерированную сеть дорог.

EntitiesState: Состояние NPC. Убитые мобы (is_dead=1).

QuestProgress: Таблица активных графов.

6.3. Система Кэширования ИИ

SaveManager ведет таблицу DialogueCache. Если хеш (ID NPC + стейт) найден в БД, текст берется локально (0 мс задержка, 0 потраченных токенов).

7. UX ARCHITECTURE & ДЕТАЛИЗАЦИЯ ИНТЕРФЕЙСА

7.1. Структура UI Сцен в редакторе (.tscn)

Интерфейсные скрипты из папки UI/ привязываются к визуальным элементам (сценам типа Control). Ожидается строгая иерархия узлов:

DialogBox.tscn: Узел ScrollContainer, внутри него Label (названный ChatHistory с autowrap_mode = Word). Отдельно Panel (InputPanel), содержащая LineEdit и Button (SendButton).

CharSheetUI.tscn: Корневой контрол со скриптом, содержащий Label с именем LevelLabel, контейнер StatsContainer с дочерними элементами (например, STRLabel).

Идентичная древовидная структура применяется к InventoryUI, CraftingUI и GlobalMap.

7.2. Масштабирование и Защита верстки

Godot Anchors & Containers: Запрет на абсолютное пиксельное позиционирование.

Anti-Overlap: Текст ИИ никогда не сдвинет соседние окна благодаря скроллу.

Resizing: Окна DraggableWindow автоматически пересчитываются пропорционально.

7.3. Асинхронные конфликты (Task Cancellation)

Передается CancellationToken. Если игрок закроет окно DialogBox не дождавшись ответа, токен отменит запрос.

7.4. Навигация: Процедурные Карты и Память Маршрутов

Отрисовка окружения использует исключительно Code-Only подход.

LocalMap (Внутренняя карта локации): Представляет собой несколько строений/зон (визуализируются как прямоугольники — ColorRect или Panel), соединенных дорогами/маршрутами (визуализируются как прямые линии через _draw() или тонкие прямоугольники).

Генерация и Память: Местоположение домов, зон и внутреннее расположение дорог между ними не захардкожено. Когда игрок впервые посещает локацию (будь то королевство, город или деревня), скрипт LocalMap алгоритмически раскидывает зоны по холсту и прокладывает между ними маршруты. Сразу после этого сгенерированная топология (Vector2 координаты зон и связи-дороги) записывается в базу данных (SQLite) и навсегда закрепляется за этой локацией. При следующих входах местоположение дорог и строений будет загружено из БД и останется прежним.

GlobalMap (Мир): Поддержка Zoom (колесиком) и выпадающего списка-поиска по открытым локациям.

7.5. Меню Персонажа (CharSheetUI)

Детально отражает состояние "Скелета": Основные статы (STR/DEX), Вторички (Уворот/Мент.Сопр), Активные травмы и баффы с таймерами.

7.6. Инвентарь, Tooltips и Меню Крафта

Вкладки с фильтрами, Hover Compare для шмота, дерево рецептов с подсветкой нехватки ингредиентов красным цветом.

8. PRODUCTION & ERROR HANDLING

ErrorManager: Circuit Breaker. Если API дает 429/500 ошибки — включается Exponential Backoff. Максимум 3 попытки, затем Fallback-заглушка.

CostTracker: Локальный токенизатор считает расходы.

TelemetryLogger: Сброс "галлюцинаций" в локальный лог.

10. APPENDIX A: BASE CONTENT COMPENDIUM (DATA-DRIVEN JSON)

10.1. Характеристики персонажа (StatManager.cs)

Сила (STR): Базовый урон ближнего боя. Пробитие блока. Лимит веса.

Ловкость (DEX): Инициатива. Точность. Урон кинжалов/луков. Шанс на побег (Flee).

Выносливость (END): Множитель Здоровья и Усталости. Сопротивление травмам.

Интеллект (INT): Максимальная Мана. Магический урон. Шанс крафта. (При INT=0 ИИ-текст меняется на руны).

Удача (Luck): Шанс Crit Success. Снижение Crit Fail. Улучшение генерации лута.

Карма: От -100 (зло) до 100 (добро). Влияет на реакцию NPC.

Ментальное сопротивление (Mental Res): Защита от контроля и скрытых ИИ-обманов.

Харизма: Ветка намерений (Убеждение, Торговля, Устрашение). Скидки.

Здоровье (HP): (END * 5) + (STR * 2). Падение до 0 = Обморок.

Усталость (Stamina): (END * 5) + (DEX * 2). Тратится на физику.

Мана (Mana): (INT * 5). Магический ресурс.

Уворот (Evasion): (DEX * 1.5) + (Luck * 0.2). Пассивный уклонист.

Кардио (Cardio): END + (STR * 0.5). Скорость регенерации Усталости.

11.4. Квестовые Цепочки (Quest Chains)

Движок (QuestManager.cs) оперирует исключительно математическими графами (Скелетами). ИИ получает эти графы и генерирует литературный контекст диалогов.

Основная цепочка: "Тень над Гаванью" (Main Quest)

Вводит игрока в конфликт деревни и бандитов, обучая основам дипломатии, боя и исследований.

Узел 1 (Триггер: Разговор с Бромом)

Скелет (Движок): [check: null] -> [action: accept_quest] -> [next_node: 2].

Промпт для ИИ: "Староста Бром просит игрока расследовать нападения волков на опушке леса. Он нервничает и недоговаривает".

Узел 2 (Бой и Улики на Опушке)

Скелет: [objective: kill_mob (wolf_01, count: 3)] -> [reward: spawn_item (item_bandit_letter)] -> [next_node: 3].

Логика: Движок спавнит 3-х волков. Бой через CombatManager. После убийства в луте генерируется "Письмо с печатью Клыка" (доказательство сговора Старосты и бандитов). ИИ генерирует текст записки.

Узел 3 (Конфронтация со Старостой)

Скелет: Игрок возвращается к Брому с письмом. Выбор (Намерения):

Опция А (Шантаж): [check: Charisma/Intimidation > 12] -> [success_reward: 50g, Karma -10] -> [next_node: 4].

Промпт для ИИ: Игрок показывает письмо. Если игрок выбрал Шантаж и прошел чек — Бром в панике отдает золото. Если Правосудие — Бром умоляет о пощаде, прежде чем его арестует Вейн.

Узел 4 (Оборона деревни)

Скелет: [event: spawn_raid (bandit_rookie, count: 4, location: zone_square)].

Логика: Бандиты узнают о срыве сделки и нападают на деревню. Включается массовый бой. Помощь кузнеца Торна и стражи зависит от прошлых действий игрока.

11.5. Торговый Город: Ривергард (Riverguard) - Расширение

Как только игрок выполняет Основную цепочку в Гавани, он получает достаточно уровня и экипировки, чтобы пережить путешествие по Глобальной карте до Ривергарда.

Особенность: При входе в город WorldBuilder (в фоне) генерирует политическую фракционную борьбу (например, Гильдия Купцов против Коррумпированной Стражи).

Масштаб: В Ривергарде генерируется 15-20 NPC, 3 района (Трущобы, Рынок, Верхний Город) и активируются сложные проверки Харизмы и Дресс-кода (в лохмотьях игрока не пустят в Верхний Город).
