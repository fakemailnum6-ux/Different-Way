1. INTRODUCTION AND PROJECT GOALS

Цель данной технической спецификации — проектирование архитектуры для текстовой RPG-песочницы с открытым миром, глубокой социальной симуляцией и интеграцией LLM (ИИ).

Данный документ представляет собой полную, развернутую архитектуру, фиксирующую протоколы общения с API нейросетей, детальную структуру хранения данных, жесткое ограничение на визуальное оформление, полный базовый справочник контента (Скелета), а также пошаговые инструкции по сборке сцен в редакторе Godot.

1.1. Key Architectural Principles

Парадигма "Скелет и Мясо" (Strict Isolation): Жесткое разделение математической логики и нарративного оформления.

Code-Only Rendering (Без внешних ассетов): На этапе разработки и в базовой версии игры полностью отсутствуют спрайты, текстуры, картинки и 3D-модели. Визуализация (карта, интерфейс, иконки) создается исключительно программным кодом Godot (использование примитивов ColorRect, Panel, RichTextLabel, процедурная генерация через _draw(), а также ASCII/UTF-8 символы для иконок).

Двухуровневая генерация (Macro & Micro): Контент генерируется в фоне (Macro) и JIT (Micro). Мир и стартовые условия полностью процедурны.

Zero-Trust AI & Fallback First: Защита от мусорных ответов и падений API. Игра никогда не доверяет данным от ИИ без валидации.

Thread-Safe Determinism: Фоновая симуляция полностью изолирована от UI-потока.

Zero-Allocation Policy: Оптимизация Garbage Collector (C#).

Data-Driven (Отвязка контента от кода): Контент живет в JSON-файлах. Базовые элементы Скелета (предметы, мобы) используются ИИ как кирпичики для процедурной сборки мира.

Direct Control (Прямое управление): Наличие классической консоли разработчика для прямого вмешательства в симуляцию в обход нейросети.

3. CORE MODULE & GODOT INTEGRATION

3.1. Теневая симуляция (Thread Safety & State Sync)

Для предотвращения рассинхронизации (desync) между фоновым потоком симуляции и потоком отрисовки UI (Godot Main Thread), архитектура использует однонаправленный поток данных (Unidirectional Data Flow), строгую иммутабельность и очередь команд.

Мутабельное ядро (GameState_Live): Текущее состояние игры (инвентарь, HP, координаты, GOAP-цели NPC). Существует в единственном экземпляре внутри потока Simulation.cs. UI не имеет ссылок на эти объекты и не может их напрямую читать или изменять.

Иммутабельные Снапшоты (GameState_Snapshot): Раз в тик симуляции Simulation.cs создает глубокую, полностью неизменяемую копию состояния (с использованием C# record типов и IReadOnlyList). Этот снапшот атомарно передается в UI поток через атомарную подмену ссылки (Interlocked.Exchange).

Версионирование (Versioning): Каждая успешная мутация в GameState_Live увеличивает счетчик StateVersion (целое число). Снапшот содержит этот Version. Компоненты UI хранят свой LastRenderedVersion и перерисовываются только если Snapshot.Version > LastRenderedVersion. Это гарантирует, что интерфейс не будет моргать или выполнять лишнюю работу.

Синхронизация через Очередь Команд (Command Queue): Если игрок нажимает кнопку (например, "Выпить зелье" или "Атаковать"), UI не меняет состояние напрямую. Вместо этого он создает объект ICommand и помещает его в потокобезопасную очередь ConcurrentQueue<ICommand>. Поток симуляции в начале каждого тика:

Вычитывает все команды из очереди.

Валидирует и применяет их к GameState_Live.

Инкрементирует StateVersion.

Публикует новый GameState_Snapshot.

Эта парадигма полностью исключает состояния гонки (Race Conditions) и десинк между математикой и визуалом.

3.2. Zero-Allocation

При формировании промптов конкатенация строк через + запрещена. Используются пулы и StringBuilder.

3.3. Настройка Autoloads (Синглтонов) в Godot

Для связи C# бэкенда с Godot и глобальной работы всех систем необходима строгая регистрация Autoloads в редакторе (Project -> Project Settings -> Autoload):

GameManager (узел из GameManager.cs). Важность: Метод _Ready() при старте игры автоматически вызывает DataManager.Initialize(), загружая все JSON-файлы (мобов, оружие, статы) в оперативную память.

Simulation (узел из Simulation.cs).

EventBus (узел из EventBus.cs).

Logger (узел из Logger.cs). Глобальный регистратор событий.

3.4. Визуализация (Code-Only)

Отсутствие TextureRect с загруженными изображениями (png/jpg). Вся графика строится на темах Godot (Theme), узлах контейнеров и отрисовке линий.

3.5. Логирование и Консоль разработчика (Direct Control)

Игра включает мощную систему отладки и управления в обход нейросети.

Logger.cs: Регистрирует все события ядра, ответы API, ошибки и переходы состояний. Данные пишутся асинхронно в файлы user://logs/game_log.txt (с ротацией файлов по дням). Уровни логов: INFO, WARNING, ERROR, AI_DEBUG.

ConsoleManager.cs: Ядро обработки текстовых команд. Поддерживает регистрацию методов через атрибуты C# (например, [ConsoleCommand("give_gold")]). Позволяет напрямую изменять переменные в GameState_Live, минуя LLMClient и GraphValidator.

3.6. Глобальный конечный автомат (Game Loop / State Machine)

Управление высокоуровневым потоком игры строго централизовано через GameStateMachine.cs. Прямые переходы между UI-компонентами (например, скрипт LocalMap скрывает себя и вызывает показ CombatUI) категорически запрещены.

Состояния (States):

State_Exploration: Активны GlobalMap или LocalMap. Работает фоновая симуляция, генерация случайных встреч, NPC выполняют GOAP-рутину.

State_Dialogue: Открыт DialogBox. Блокируется перемещение, время (TimeManager) приостанавливается или сильно замедляется, ИИ обрабатывает запросы.

State_Combat: Открыт CombatUI. Включен жесткий пошаговый таймер боя. Вызовы ИИ заблокированы (кроме fallbacks). Делегирует математику локальному автомату CombatManager.

State_Menu: Игрок открыл Инвентарь, Меню персонажа или Паузу. Теневая симуляция приостановлена (или работает в фоне без влияния на позицию игрока).

Транзиции (Transitions):
Все переходы происходят исключительно по сигналам шины. Например, если игрок атакует NPC в диалоге, DialogBox делает EventBus.Emit("RequestStateChange", GameState.Combat). GameStateMachine перехватывает сигнал, корректно скрывает окно диалога, приостанавливает LLMClient и инициализирует экран боя, гарантируя отсутствие визуальных конфликтов и наложений (overlays).

3.7. Единый Игровой Цикл (The Unified Game Loop)

Для предотвращения спагетти-кода и рассинхронизации, вся архитектура работает как единый однонаправленный конвейер (Pipeline). Ни один UI-элемент или ИИ-ответ не меняет игру напрямую.

Схема цикла (Tick):
Input -> Intent -> Simulation -> EventBus -> UI

Input (Ввод): Игрок кликает по кнопке (перейти на локацию, атаковать, отправить сообщение в чат). Слой UI ничего не вычисляет, он лишь формирует структуру ICommand (например, CommandMoveToNode, CommandTalkToNpc).

Intent (Намерение): Команда помещается в потокобезопасную очередь ConcurrentQueue. Важное уточнение: Если это был текстовый ввод в чат, LLMClient асинхронно превращает текст в JSON-ответ, и этот ответ парсится в ICommand (например, CommandGiveItem), который точно так же ложится в очередь.

Simulation (Симуляция): В начале каждого тика фоновый поток извлекает команды из очереди. GameState_Live проверяет их на легальность (хватает ли стамины, жив ли враг). Если проверка пройдена, происходит детерминированная мутация данных.

EventBus (События): Сразу после изменения данных (урон получен, квест взят), ядро испускает сигналы: EventBus.Emit("OnHealthChanged"), EventBus.Emit("OnStateTransition").

UI (Отрисовка): Главный поток (Godot Main Thread) подхватывает обновленный GameState_Snapshot (с инкрементированной Version) и реагирует на сигналы EventBus, обновляя визуал.

4. ИГРОВЫЕ СИСТЕМЫ, ДАННЫЕ И МЕХАНИКА (THE SKELETON)

4.1. Боевая система (Combat Manager State Machine)

Бой полностью изолирован от нейросети. Работает в собственном окне UI (CombatUI.cs). Случайность дает лишь вариативность (±10% урона) и шанс критического удара. Сам класс CombatManager реализует строгий вложенный конечный автомат.

Локальные состояния боя (Combat States):

CombatState.Init: Расчет стартовой инициативы (сортировка очереди всех участников). Применение предбоевых баффов. Позиционирование (Авангард/Арьергард).

CombatState.PlayerTurn: Запуск жесткого таймера (10 сек). Ожидание ввода игрока (можно сделать 1 Основное действие — Удар/Магия, и 1 Свободное — Выпить зелье). Если таймер истекает без ввода — принудительный переход в Resolve со статусом "Авто-защита".

CombatState.EnemyTurn: Ход ИИ-противника. Происходит мгновенно. Противник выбирает цель (всегда сперва Авангард), проверяет кулдауны спец-навыков. Если HP < 15% и нет трейта "Бесстрашный", враг попытается применить "Побег".

CombatState.Resolve: Этап применения математики (см. 4.1.1). Расчет бросков на попадание/уклонение. Вычет брони из урона. Изменение полосок HP. Применение тиков от статусных эффектов (Яд, Горение). Проверка условий смерти участников.

CombatState.End: Окончание боя. Подсчет наград, вызов ItemGenerator для дропа, начисление золота/штрафов. Отправка сигнала в GameStateMachine для закрытия UI и возврата в State_Exploration.

Локальные Транзиции (Переходы):
Цикл боя: Init -> выбор первого по инициативе (PlayerTurn или EnemyTurn) -> выполнение действия -> Resolve.
Из Resolve автомат проверяет условия победы/поражения. Если обе стороны живы -> переход к следующему живому участнику в очереди (PlayerTurn или EnemyTurn). Если одна сторона мертва (или сбежала) -> переход в End.

4.1.1. Формальная математика боя (DamageCalculator.cs)

Все расчеты производятся в фазе Resolve и строго задокументированы, чтобы исключить "хардкод" и магические числа.

Инициатива (Очередь ходов):

Initiative = DEX + Random(1, 10)

Сортировка по убыванию. Рассчитывается один раз в фазе Init.

Шанс попадания (Hit Chance & Evasion):

Базовая точность (Base Accuracy): 80%

Бонус атакующего (Attacker Accuracy): (Attacker_DEX * 1.5) + (Attacker_Luck * 0.2)

Уворот защитника (Defender Evasion): (Defender_DEX * 1.5) + (Defender_Luck * 0.2) (См. пункт 10.1)

Формула: HitChance = Base (80) + Attacker Accuracy - Defender Evasion

Ограничитель (Clamp): Шанс попадания не может быть ниже 5% и выше 100%. Роллится случайное число RNG(1, 100). Если RNG <= HitChance, атака успешна.

Шанс критического удара (Crit Formula):

Базовый шанс (Base Crit): 5%

Бонус от Удачи: +(Attacker_Luck * 1%)

Модификаторы оружия: У кинжалов дополнительно +2%.

Формула: CritChance = 5 + (Luck * 1) + Weapon_Crit_Bonus. Роллится RNG(1, 100).

Множитель урона (Crit Multiplier): По умолчанию 1.5x.

Критический промах (Crit Fail): При выпадении RNG(1, 100) <= 2. Атака прерывается, оружие получает двойной износ.

Расчет урона (Damage Calculation):

Базовый урон оружия (Weapon Damage): Фиксированное число из JSON.

Скейлинг от характеристик (Stat Scaling):

Мили (Мечи/Дубины): ScaleBonus = (Attacker_STR * 0.5)

Рендж/Кинжалы: ScaleBonus = (Attacker_DEX * 0.5)

Магия: ScaleBonus = (Attacker_INT * 0.5)

Вариативность (Variance): ±10% от базового урона.

Формула "Грязного" Урона (Raw Damage):
RawDamage = (WeaponDamage + ScaleBonus) * Random(0.9, 1.1) * (IsCrit ? 1.5 : 1)

Поглощение броней (Armor Reduction):

Броня работает как линейное поглощение (Flat Reduction), а не процентное. Каждая 1 единица Брони блокирует 1 единицу Урона.

Формула "Чистого" Урона (Final Damage):
FinalDamage = Max(1, RawDamage - Defender_Armor)

Ограничитель: Успешное попадание всегда наносит минимум 1 урон, если только не заблокировано специальным эффектом (например, "Неуязвимость").

Побег (Fleeing): Доступен как действие в ход игрока/врага.

Математическая проверка: (DEX + Cardio) убегающего > (DEX + Cardio) самого быстрого преследователя.

Провал означает пропуск хода и удар в спину (Attacker HitChance = 100%). Успех досрочно переводит автомат в CombatState.End.

4.2. Состояние проигрыша (Death & Failure State)

Падение здоровья (HP) до 0 приводит к нарративному сдвигу и экономическому штрафу.

Механика Respawn: Simulation проматывает игровое время на 1-3 дня вперед. Персонаж "просыпается" в ближайшей безопасной зоне (Храм/Таверна).

Штрафы: Потеря % золота, резкое снижение прочности надетой экипировки, получение "Травмы" (дебафф статов).

Интеграция ИИ: При пробуждении LLMClient генерирует короткий текст о том, как вас спасли.

4.3. Износ экипировки (Durability)

В DataManager внедрены параметры CurrentDurability и MaxDurability.

Износ: Снижение прочности оружия при атаках (особенно по бронированным целям), брони — при получении урона.

Поломка: При достижении 0 предмет не исчезает, но его параметры обнуляются (до 1), пока он не будет восстановлен у кузнеца.

4.4. Глобальная симуляция, Маршруты и Экономика (Работа)

Мир — это взвешенный граф со строгой архитектурной вложенностью. Навигация происходит от общего к частному:

Королевство (Kingdom): Макро-регион, содержащий сеть поселений. Отображается на GlobalMap.

Поселение (Город/Деревня): Набор строений (домов, магазинов, гильдий, ферм), алгоритмически сгенерированных и соединенных локальными дорогами на экране LocalMap.

Здание/Дом/Точка Интереса (Building/POI): Конкретная точка (прямоугольник на карте) внутри поселения. Также включает в себя места базового заработка.

Меню Взаимодействия (Interaction Node): При заходе внутрь здания игрок не управляет персонажем напрямую (нет брождения внутри), а попадает в текстово-кнопочное меню. Здесь отображается:

Список доступных NPC: Имена и роли персонажей (генерируется GOAPEngine). Кликом GameStateMachine переводится в State_Dialogue.

Список Действий (Actions): Интерактивные возможности. Например:

В таверне: "Снять комнату", "Начать торговлю".

На ферме / В шахте: Базовый заработок (см. ниже).

Базовая экономика и Гринд (Работа):
Чтобы игрок не застрял без золота на старте, существуют специальные узлы (Фермы, Лесопилки, Шахты). При входе в них появляется опция:
Отработать смену: 8 часов
Логика движка: При нажатии расходуется 80% шкалы Stamina, время TimeManager проматывается на 8 часов вперед, и игрок гарантированно получает фиксированную сумму золота (например, 10g) без участия боевой системы или ИИ.

4.5. Система квестов и их выдачи (Quest Pipeline)

Сборка Скелета: QuestManager собирает математический граф. Пример: [Node A: Идти в Пещеру] -> [Node B: Убить Бандита] -> [Node C: Получить Золото].

Генерация Мяса: Скелет отправляется в ИИ. ИИ возвращает JSON с названием, лорной причиной и текстом для NPC.

Исполнение и Перемещение: После принятия квеста, цель (целевой узел, например "Логово Волков") подсвечивается на GlobalMap или LocalMap. Игроку необходимо физически кликнуть по маршруту и добраться до места выполнения.

Если цель квеста "Убить моба", бой не начнется прямо в таверне. Игроку нужно дойти до узла обитания моба (например, zone_forest_edge), после чего триггерится CombatManager.

QuestManager локально следит за счетчиком прогресса (убитые волки, собранные травы) и обновляет статус квеста.

4.6. Крафт, Поведение NPC (GOAP) и Конфликт с LLM

CraftingEngine: Валидирует наличие ингредиентов. Рассчитывает шанс успеха на основе Интеллекта. При успехе запрашивает у ИИ название и описание предмета.

GOAPEngine: Математически рассчитывает рутину NPC (Голод = 80 -> Цель: Поесть). Результат передается в ИИ как контекст для логичных ответов в диалоге.

Конфликт вероятностей (Принцип Иммутабельности Намерений):
GOAP — строго детерминирован, а LLM — вероятностен. Возможна ситуация, когда NPC обещает пойти в бой, но по GOAP он идет спать. Чтобы предотвратить этот десинк, вводится Запрет на изменение намерений. LLM не имеет права менять текущую цель (Intent) NPC. Функция LLM сводится к генерации литературного обоснования детерминированной математики (например: "Прости, путник, я валюсь с ног...").

Протокол Эскалации и Защита от спама (Patience System):
Чтобы игрок не мог бесконечно "докапываться" до спящего или занятого NPC, ломая логику мира, вводится система Терпения.

Если игрок начинает/продолжает диалог с NPC, чей GOAP-статус имеет высокий приоритет (Спит, Бежит, Работает), в промпт P0 передается инструкция: [SYSTEM: Ты занят. Если игрок настаивает, используй триггер "end_dialogue"].

ИИ генерирует раздраженный/короткий ответ и возвращает в JSON {"type": "end_dialogue"}. Это заставляет GameStateMachine принудительно выбросить игрока из состояния State_Dialogue.

Движковый Кулдаун (Fallback): Как только диалог закрыт триггером end_dialogue, Simulation вешает на этого NPC временный блок (например, 30 секунд реального времени). Если игрок попытается заговорить с ним снова до истечения таймера, API-запрос к ИИ не отправляется. Движок напрямую выводит Floating Text: "NPC вас игнорирует". Если у NPC прописан агрессивный характер, движок может инициировать State_Combat.

4.7. Универсальная Генерация Локаций (WorldBuilder Pipeline)

Чтобы обеспечить бесшовный и единообразный опыт, создание любой новой локации — будь то самая первая стартовая деревня (Протокол Нулевого Состояния) или случайно открытый город на 50-м часу игры — осуществляется через единый универсальный метод WorldBuilder.GenerateLocation(). Этот алгоритм гарантирует, что все локации подчиняются одинаковым правилам физики, экономики и топологии.

4.7.1. Входные данные генератора (Inputs)

Функция GenerateLocation(LocationParams) принимает жесткий набор параметров (Скелет), которые задают рамки для ИИ:

Biome (Биом): Лес, Пустошь, Горы, Болото (определяет пулы мобов на окраинах и тип ресурсов).

Type (Тип поселения): Столица, Торговый город, Деревня, Лагерь, Руины (определяет количество зданий и уровень цен).

Faction (Контекст фракции): Кто контролирует зону (Королевство, Бандиты, Нейтралы). Влияет на стражу и законы.

PlayerLevel (Уровень игрока): Используется для скалирования параметров локальных подземелий, таблиц лута в сундуках и уровня торговцев.

NarrativeContext (Лорный контекст): Текстовая строка. (Для старта игры сюда передается предыстория игрока; для мид-гейма — текущие глобальные события, например "В этом регионе идет война").

4.7.2. Процесс генерации (The Pipeline)

Обращение к ИИ (Мясо): WorldBuilder отправляет Inputs в LLMClient. ИИ возвращает JSON с названием города, общим лором, массивом зданий (Таверна, Кузница) и массивом NPC (для каждого из которых ИИ сам генерирует GOAP-цели и подбирает логичный инвентарь из базы /Data).

Алгоритмическая топология (Математика): Движок берет массив зданий и алгоритмически расставляет прямоугольники (ColorRect) на экране LocalMap. Затем он строит граф — прокладывает прямые линии (дороги) между зданиями, следя за тем, чтобы граф был связным.

Персистентность (Сохранение): Сгенерированные координаты X/Y каждого здания, граф дорог, списки NPC и их инвентари записываются в SQLite БД (WorldNodes).

Завершение: Генерация завершена.

4.7.3. Выходные данные генератора (Outputs)

На выходе WorldBuilder.GenerateLocation() выдает жестко структурированный узел в памяти:

Название и Описание локации.

Фиксированный граф карты: При любом последующем входе в эту локацию, LocalMap читает координаты из БД, и дороги/дома стоят ровно на тех же местах, что и при генерации.

Списки зон: Полноценно функционирующие здания (Таверны с торговцами, Опушки с привязанными спавнерами мобов [wolf_01, bear_01]).

4.8. Механика Статусных Эффектов (Stacking & Refresh Rules)

Для предотвращения поломки баланса (бесконечный стак ядов или перманентный стан), StatusEffectManager.cs использует строгие правила разрешения конфликтов:

Независимое сосуществование (Coexistence): Разные типы эффектов (например, Яд и Кровотечение) работают параллельно и наносят урон раздельно в фазу Resolve.

Правило обновления (Refresh Rule): Если на цель накладывается эффект, который уже активен и не имеет свойства Stackable (например, повторное Отравление), его сила не суммируется. Таймер длительности (Duration) просто сбрасывается до максимума.

Правило суммирования (Stacking): Уникальные эффекты (обозначены тегом [Stackable]) могут складываться. Например, Кровотечение: каждый новый удар добавляет стак (до максимума в 3 стака). Урон умножается на количество стаков, таймер обновляется.

Иммунитет к контролю (CC Diminishing Returns): Эффекты жесткого контроля (Оглушение) после окончания накладывают скрытый бафф "Сопротивление контролю" на 1 ход, делая цель иммунной к повторному оглушению на следующий ход (защита от станлока).

Взаимоисключение (Mutual Cancellation): Противоположные эффекты уничтожают друг друга. Наложение эффекта "Горение" на цель со статусом "Намокание" снимает оба эффекта (результат: пар, урона нет).

5. AI MANAGEMENT & API INTERACTION (ИНТЕГРАЦИЯ REST API)

В качестве основного API-провайдера для генерации лора, квестов и ответов NPC используется платформа Puter с моделью grok-4-fast. Взаимодействие реализовано через HTTP REST запросы, полностью совместимые с форматом сообщений OpenAI.

5.1. Формирование запроса (API Payload Builder)

ИИ работает как слой генерации поверх математического "Скелета". Запросы обрабатываются в модуле LLMClient.cs через класс HttpClient (или HTTPRequest в контексте Godot Engine).

Точка доступа (Endpoint) и Заголовки (Headers):

URL: https://api.puter.com/puterai/openai/v1/chat/completions

Method: POST

Headers: * Authorization: Bearer <API_TOKEN>

Content-Type: application/json

Структура тела запроса (Payload):
Запрос формируется в виде JSON, где системный промпт (P0) и игровой контекст (P1-P3) разделены по ролям system и user.

// Пример концептуальной реализации в LLMClient.cs
var requestBody = new {
    model = "grok-4-fast",
    stream = false, // Можно переключить на true для потокового вывода
    messages = new[] {
        new { role = "system", content = "Твоя цель: отыгрывать NPC. Верни строго валидный JSON." },
        new { role = "user", content = promptPayload } // Контекст игры, статы, реплика игрока
    }
};

string jsonBody = JsonSerializer.Serialize(requestBody);
// Отправка через HttpClient...


Ожидаемый ответ от ИИ (JSON Schema):
Системный промпт жестко инструктирует модель обертывать свои ответы в JSON-структуру. После получения ответа от сервера, текст извлекается по стандартному пути data.choices[0].message.content.

{
  "thoughts": "Игрок манипулирует. Репутация высока. Я соглашусь.",
  "spoken_text": "Держи кружку. Но это в последний раз!",
  "action_triggers": [
    {"type": "give_item", "id": "drink_ale", "amount": 1},
    {"type": "end_dialogue"}
  ]
}


(Примечание: Если в запросе установлен параметр stream: true для эффекта "печати текста" в реальном времени, LLMClient должен читать потоковые HTTP-ответы (Server-Sent Events) и парсить choices[0].delta.content, накапливая финальную строку).

5.2. Защита от Prompt Injection (InputSanitizer.cs)

Санитаризация: Ввод прогоняется через blacklists и RegEx (вырезаются мета-команды ignore instructions, System:).

System Prompt Armor: Системные инструкции (P0) требуют игнорировать команды из блока [Player_Input].

5.3. Базовая Валидация JSON и Данных (Validation Pipeline)

Игра никогда не исполняет команды из action_triggers вслепую. Для предотвращения overengineering на старте, ответ ИИ проходит через упрощенный конвейер валидации в классе GraphValidator.cs.

Уровень 1: Синтаксическая очистка и парсинг (Syntax Check & Schema DTO)

Модели ИИ часто оборачивают ответ в markdown-блоки (например, ```json { ... } ```) или добавляют лишний текст.

GraphValidator ищет первый { и последний }, извлекая чистую строку.

Происходит парсинг через System.Text.Json в C#-модель данных. Обязательны поля thoughts и spoken_text. Если парсинг падает, запрос отбраковывается.

Уровень 2: Семантическая сверка со Скелетом (Data Registry Match)

Самый важный этап. ИИ склонен к галлюцинациям и может попытаться "подарить" игроку предмет, которого не существует.

Если ИИ выдает команду {"type": "give_item", "id": "magic_excalibur_99", "amount": 1}, GraphValidator проверяет id по словарю DataManager (загруженному из файлов /Data).

Так как magic_excalibur_99 отсутствует в res://Data/Items/Weapons.json, команда вырезается (игнорируется), а в TelemetryLogger пишется предупреждение. Текст диалога (spoken_text) при этом выводится игроку, но предмет не выдается.

Обработка ошибок (Fallback)

В целях оптимизации разработки сложные экономические лимиты и циклы повторных запросов (Retry Loops) временно отключены.

Fallback: Если JSON критически сломан (не прошел Уровень 1), ИИ ответ игнорируется, и игроку выводится локальная заглушка (Fallback-строка из Data/Localization), например: "NPC задумчиво молчит, словно забыв слова." Никаких крашей UI или вылетов игры не происходит.

5.4. Память ИИ: Контекст диалога (FIFO) и планы на RAG

Для минимизации издержек и упрощения отладки на этапе MVP в игре используется скользящее окно (FIFO) вместо сложной и ресурсоемкой базы векторного поиска (VectorDB). Игрок взаимодействует с NPC короткими сессиями, поэтому хранение последних нескольких сообщений полностью покрывает нужды геймплея.

Реализация памяти на старте (FIFO-очередь): Модуль ContextManager хранит только 10-15 последних сообщений из текущей беседы. Старые сообщения выталкиваются из очереди, чтобы экономить токены и не перегружать контекстное окно. Собеседник помнит текущую тему разговора, но "забывает" то, о чем игрок спрашивал его неделю назад.

Итоговое распределение токенов промпта (Фаза 1):

System Rules (P0): ~300 токенов. (Правила формата JSON и запрет на изменение GOAP-намерений).

Current Objective & GOAP (P1): ~200 токенов.

World State (P2): ~500 токенов. (Время, текущая локация).

Dialogue History (P3 - FIFO): Оставшиеся токены на прямой лог текущей беседы.

Планы на будущее (Фаза 2): Внедрение VectorDB.cs (RAG) для семантического поиска воспоминаний (например, "Игрок украл у меня яблоко 2 дня назад") отложено до момента, когда Core Loop (Фаза 1) будет полностью стабилизирован.

6. DATA PERSISTENCE & STORAGE ARCHITECTURE (ХРАНЕНИЕ ДАННЫХ)

6.1. Пользовательские директории (Godot user://)

user://saves/ — БД сохранений.

user://cache/ — Кэш ИИ.

user://exported_worlds/ — Префабы (Скелеты макро-мира) в формате JSON.

user://config/ — Настройки UI, ключи (зашифрованные).

user://logs/ — Текстовые логи игры, генерируемые Logger.cs.

6.2. База данных сохранений (SQLite)

Каждое сохранение — это файл .db. Позволяет подгружать мир чанками.

PlayerState: Базовые статы, уровень, координаты, время.

Inventory: Таблица связи [player_id] -> [item_id] -> [quantity] -> [current_durability].

WorldNodes: Состояние открытых локаций, включая их внутреннюю топологию (координаты прямоугольников) и сгенерированную сеть дорог.

EntitiesState: Состояние NPC. Убитые мобы (is_dead=1).

QuestProgress: Таблица активных графов.

6.3. Система Кэширования ИИ

SaveManager ведет таблицу DialogueCache. Если хеш (ID NPC + стейт) найден в БД, текст берется локально (0 мс задержка, 0 потраченных токенов).

7. UX ARCHITECTURE & ДЕТАЛИЗАЦИЯ ИНТЕРФЕЙСА

Основная концепция: вся информация доступна прямо с экранов карты без "закопанности" в многоуровневых подменю. Управление экранами координируется через GameStateMachine.

7.1. Структура UI Сцен в редакторе (.tscn)

Интерфейсные скрипты из папки UI/ привязываются к визуальным элементам (сценам типа Control). Ожидается строгая иерархия узлов:

HUDManager.tscn: Оверлей, видимый всегда поверх карт. Отображает Золото, время, полоски HP/Маны и закрепленный виджет ActiveQuestTracker.

DialogBox.tscn: Узел ScrollContainer с Label (ChatHistory, autowrap_mode = Word). Благодаря интеграции stream: true из Puter API, текст здесь появляется с динамическим "эффектом печати" по мере получения чанков от ИИ. Отдельно Panel (InputPanel), содержащая LineEdit и Button (SendButton).

QuestJournalUI.tscn: Окно с тремя вкладками: "Активные", "Завершенные", "Проваленные". Клик по квесту устанавливает его как "отслеживаемый" для HUD.

CombatUI.tscn: Разделенный на две половины экран (Авангард/Арьергард). Справа/Слева панели состояний, внизу кнопки действий и лог битвы.

CharSheetUI.tscn, InventoryUI.tscn, GlobalMap.tscn, LocalMap.tscn строятся по аналогичному принципу на базе MarginContainer.

7.2. Внутриигровая консоль разработчика (ConsoleUI.tscn)

Реализуется классическая консоль в стиле движка Quake (выпадает сверху экрана).

Активация: По умолчанию привязана к клавише Тильда (~) в InputHandler.cs.

Функционал: Отображает прямой вывод из Logger.cs. Позволяет разработчику (или игроку в режиме песочницы) вводить строковые команды, которые парсятся ConsoleManager.cs.

7.3. Масштабирование и Защита верстки

Godot Anchors & Containers: Запрет на абсолютное пиксельное позиционирование. Все элементы привязаны к краям экрана.

Anti-Overlap: Текст ИИ никогда не сдвинет соседние окна. При переполнении контейнера появляется ползунок прокрутки.

Resizing: Окна автоматически пересчитывают свои пропорции при изменении разрешения игры.

7.4. Асинхронные конфликты (Task Cancellation)

При запросах к puter.ai.chat передается CancellationToken. Если игрок закроет окно DialogBox не дождавшись окончания запроса, токен отменит HTTP-вызов, предотвращая попытку записи в выгруженный интерфейс.

7.5. Навигация: Карты, Здания и Флоу Взаимодействия

Отрисовка окружения использует исключительно Code-Only подход.

GlobalMap (Карта Королевства): Игрок видит все открытые поселения и POI. Выбор локации осуществляется простым кликом по прямоугольнику на карте, либо через выпадающий список-поиск в углу экрана. При клике GameStateMachine переключает интерфейс на LocalMap выбранного поселения.

LocalMap (Карта местности/поселения): Представляет собой несколько строений/домов (прямоугольники ColorRect), соединенных дорогами (тонкие линии/прямоугольники).

Меню Здания: Кликнув на любое строение (например, Кузницу), игрок визуально "заходит" в него. Карта перекрывается панелью Взаимодействия.

Здесь отображается список присутствующих внутри NPC (кнопка "Поговорить" -> переводит стейт в State_Dialogue) и список действий с окружением (кнопка "Взаимодействовать" -> скрафтить предмет).

7.6. Журнал Квестов (Quest Journal) и Трекер

Вся квестовая нагрузка выведена на основные экраны:

HUD Tracker: В правом верхнем углу интерфейса (над миникартой или сбоку) всегда висит виджет последнего отслеживаемого квеста с кратким требованием (например, "Охота на волков: 1/3 убито").

Quest Journal: Меню, вызываемое кнопкой "Квесты", разделенное на вкладки (Активные / Выполненные / Проваленные). Позволяет прочитать сгенерированный ИИ лор задания и переключить активный трекер.

7.7. Интерфейс Боя (Combat UI)

При встрече с врагом (на маршруте или по квесту) GameStateMachine переводит игру в State_Combat. LocalMap или GlobalMap скрываются. Появляется экран боя:

Слева: Панель Игрока (HP, Stamina, Mana, Баффы/Травмы).

Справа: Список противников с их полосками здоровья (и позиционированием: Авангард или Арьергард).

7.8. Меню Персонажа (CharSheetUI) и Инвентарь

Детально отражает состояние "Скелета": Основные статы (STR/DEX), Вторички (Уворот/Мент.Сопр), Активные травмы и баффы с таймерами. В Инвентаре используются Вкладки с фильтрами, Hover Compare для шмота (сравнение при наведении мыши), а в Меню Крафта — дерево рецептов с подсветкой нехватки ингредиентов красным цветом. Переход в меню управляется через State_Menu.

8. PRODUCTION & ERROR HANDLING

ErrorManager: Circuit Breaker. Если API дает ошибки — включается Exponential Backoff. Максимум 3 попытки, затем Fallback-заглушка.

CostTracker: Локальный трекер считает количество обращений к API.

TelemetryLogger: Сброс "галлюцинаций" в локальный лог.

11. APPENDIX B: ZERO-STATE AI GENERATION (MACRO & MICRO)

Вместо жестко заданных стартовых локаций (захардкоженных городов и NPC), игра использует Протокол Нулевого Состояния (Zero-State Protocol). Движок опирается на LLMClient и универсальный WorldBuilder, чтобы процедурно сгенерировать мир для каждого нового прохождения, используя сущности из Приложения А как строительный материал.

11.1. Создание персонажа (Zero-State Character Gen)

При запуске "Новой Игры" перед игроком появляется пустой DialogBox в режиме "Создание персонажа".

Dungeon Master (AI): ИИ (grok-4-fast) приветствует игрока и просит кратко описать свою предысторию (кто он, откуда, чем занимался).

Ввод Игрока: Игрок вводит свободный текст (например: "Я бывший городской стражник, которого выгнали за пьянство, теперь скитаюсь без гроша").

Парсинг Характеристик: LLMClient анализирует текст и возвращает JSON, который DataManager использует для инициализации PlayerState в БД:

Статы: Рассчитываются логически (стражник = высокий STR, пьянство = низкий INT/END).

Экипировка: ИИ выбирает стартовые вещи из Приложения А (например, выдает sword_iron и rags_01). Инвентарь обнуляется.

11.2. Генерация стартовой локации (Macro-World Gen)

Как только профиль игрока сформирован, движок запрашивает генерацию стартового узла через алгоритм универсальной генерации WorldBuilder.GenerateLocation(). Он собирает локацию на основе входных параметров (текущий лор, биом, уровень игрока). Эта же функция используется и далее в мид-гейме.

Генерация Сеттинга: ИИ придумывает название, лор и тип стартового поселения на основе предыстории игрока.

Сборка LocalMap: Через универсальный пайплайн генерируется массив требуемых зданий. Алгоритм отрисовки LocalMap случайным образом располагает эти зоны (прямоугольники) на экране и прокладывает между ними маршруты-линии (записывая топологию в БД).

Заселение NPC: ИИ генерирует стартовых NPC, прописывая им ID, Имя, Профессию и базовую цель для GOAPEngine.

11.3. Динамический стартовый квест (Micro Gen)

После появления на стартовой локации игрок не имеет захардкоженного мейн-квеста.

Запрос на конфликт: Движок (через QuestManager) скрытно отправляет ИИ запрос: "Сгенерируй локальный конфликт для данного поселения с учетом предыстории игрока".

Генерация Графа: ИИ возвращает математический "Скелет" квеста, используя доступных мобов из Приложения А.

Пример Скелета: [objective: kill_mob (bear_01, count: 1)] -> [reward: spawn_item (potion_hp_medium)].

Интеграция в мир: ИИ назначает одного из сгенерированных NPC "Квестодателем". При первом разговоре с этим NPC, ИИ обернет математический Скелет в литературный текст (Мясо) и предложит игроку контракт, который отобразится в QuestJournalUI.
